## Обзор

Данный документ содержит предложения по созданию системы правил, основанной на JSON-файлах, для проекта **FEONT** - приложения для работы с графом знаний. Система адаптирована из проекта FEMSQ и настроена под специфику работы с организационными структурами, техническими структурами и планами работ на основе RDF/SHACL технологий.

## Специфика проекта FEONT

### Назначение проекта

FEONT - **исследовательский инструмент** для моделирования и эволюции бизнес-процессов через граф знаний:
- Работа с организационными структурами
- Работа с техническими структурами
- Работа с планами работ
- **Динамическое создание и модификация "классов" в режиме эксплуатации**
- Изменение схемы данных в реальном времени

### Технологический стек (на основе анализа)

**Backend:**
- Apache Jena (Jena Fuseki, TDB2, SHACL)
- Java (Spring Boot - опционально, можно начать с прямого доступа к Fuseki)

**Frontend:**
- Vue 3 + Quasar Framework
- Cytoscape.js (визуализация графов данных)
- WebVOWL (визуализация онтологий)
- YASGUI (SPARQL интерфейс)
- rdflib.js (работа с RDF в браузере)

**Storage:**
- Apache Jena TDB2 (triple store)
- Named Graphs для организации данных (ontology, data, shacl, history)

**Инструменты:**
- Protégé Desktop (моделирование онтологий)
- Docker (развёртывание)

### Ключевые особенности проекта

1. **Гибкость превыше производительности** - исследовательский инструмент
2. **Изменение схемы в реальном времени** - без миграций БД
3. **Визуализация критична** - без неё система теряет восприятие
4. **Open Source компоненты** - максимальная гибкость
5. **Небольшая команда** - 2-3 человека (разработчики-пользователи)
6. **Малый масштаб данных** - десятки классов, от единиц до десятков тысяч экземпляров

### Архитектурные решения

- **RDF + SHACL** для гибкой схемы и валидации
- **Named Graphs** для организации данных и версионирования
- **SPARQL** для аналитики и запросов
- **Граф знаний** как основа архитектуры
- **Два режима UI**: технический (RDF/SPARQL) и бизнес-ориентированный

## Анализ системы FEMSQ

### Ключевые компоненты системы:

1. **`.cursorrules`** - главный файл правил для AI-ассистента
   - Контекст проекта
   - Правила кодирования
   - Правила работы с инструментами (Desktop Commander, DBHub)
   - Ссылки на документацию
   - Проверка окружения разработки

2. **`docs/project/project-docs.json`** - центральная документация проекта
   - Метаданные проекта
   - Архитектура и технологический стек
   - Модули, компоненты, классы (иерархическая структура)
   - Схема базы данных
   - Конфигурация развертывания
   - Временные рамки, риски, ресурсы

3. **`docs/development/project-development.json`** - планирование разработки
   - Словарь задач (tasks.items)
   - Иерархические структуры задач (structure.trees)
   - Поддержка DAG (задачи могут быть в нескольких местах)
   - Статусы с эмодзи для визуальной навигации
   - Критерии завершения вместо оценок времени

4. **`docs/journal/project-journal.json`** - дневник разработки
   - Хронологические записи о работе
   - Структура: chat → log → entries (действия)
   - Ссылки на задачи и артефакты
   - Метаданные (время, усилия, статус)

5. **Правила документации** - Markdown файлы с описанием:
   - `project-documentation-rules.md` - общие правила
   - `project-docs-json-rules.md` - правила для project-docs.json
   - `project-development-json-rules.md` - правила для project-development.json
   - `project-journal-json-rules.md` - правила для project-journal.json

6. **Расширения** - дополнительные JSON файлы:
   - `docs/project/extensions/modules/modules.json` - детальная структура модулей
   - `docs/project/extensions/database/*.json` - конфигурация БД
   - И другие расширения по необходимости

### Этап 2.3: Организация работы с чатами (Приоритет: Средний)

#### Структура планов и резюме чатов

В FEMSQ используется структурированный подход к планированию и документированию работы в отдельных чатах. Рекомендуется создать аналогичную структуру для FEONT:

```
docs/development/notes/chats/
├── chat-plan/              # Планы работы на один чат
│   ├── chat-plan-YYYY-MM-DD.md
│   ├── chat-plan-YYYY-MM-DD-NNN.md  # Если несколько планов в день
│   └── README.md           # Описание структуры планов (опционально)
└── chat-resume/            # Резюме выполненных чатов
    ├── chat-resume-YYYY-MM-DD.md
    └── README.md           # Описание структуры резюме (опционально)
```

#### Структура плана чата (`chat-plan/chat-plan-YYYY-MM-DD.md`)

**Обязательные разделы:**

1. **Метаданные:**
   - Дата
   - Автор
   - Связанные задачи (task:XXXX)
   - Связанные предложения/документы (опционально)

2. **Контекст:**
   - Описание ситуации перед началом работы
   - Связь с предыдущими работами
   - Цели чата

3. **Структурный план (№ XX):**
   - Иерархический список задач с номерами (например, 01.0, 01.1, 01.1.1)
   - Статусы задач (✅ завершено, ⏳ в работе, ⚠️ требует внимания)
   - Связи с задачами из project-development.json
   - Примечания о зависимостях и последовательности

4. **Контрольные точки:**
   - Список контрольных точек (K1, K2, K3 и т.д.)
   - Критерии закрытия
   - Статусы

5. **Статус плана:**
   - Общий статус выполнения
   - Детальный статус по этапам

6. **Предложения по следующим шагам:**
   - Что делать дальше
   - Рекомендации

- Чеклисты контрольных точек

- Списки задач

#### Правила именования документов

**Формат:** `YY-MMDD-название-документа.md`

**Примеры:**
- `25-1130-ontology-structure-proposal.md` - предложение по структуре онтологии от 30 ноября 2025
- `25-1130-fuseki-deployment-plan.md` - план развёртывания Fuseki от 30 ноября 2025
- `25-1201-sparql-queries-strategy.md` - стратегия работы с SPARQL запросами от 1 декабря 2025
- `25-1202-shacl-validation-checklist.md` - чеклист валидации SHACL от 2 декабря 2025

**Правила:**
- YY - последние две цифры года (25 для 2025)
- MMDD - месяц и день (1130 для 30 ноября)
- Дефис после даты обязателен
- Название документа на английском или транслитерации, строчными буквами
- Использовать дефисы для разделения слов
- Расширение `.md` обязательно

### Особенности документации графа знаний

#### 1. Документирование онтологии

В `project-docs.json` должна быть секция для описания онтологии:

```json
{
  "ontology": {
    "base_uri": "http://example.org/feont/",
    "version": "0.1.0",
    "lastUpdated": "2025-11-30",
    "main_classes": [
      {
        "uri": "feont:OrganizationalStructure",
        "label": "Организационная структура",
        "description": "...",
        "properties": [...]
      }
    ],
    "shapes": [
      {
        "uri": "feont:OrganizationalStructureShape",
        "targetClass": "feont:OrganizationalStructure",
        "properties": [...]
      }
    ]
  }
}
```

#### 2. Документирование SPARQL запросов

В расширении `sparql/queries-library.json`:

```json
{
  "queries": {
    "analytical": [
      {
        "id": "q001",
        "name": "Найти все технические системы без резервирования",
        "sparql": "SELECT ?system WHERE { ... }",
        "description": "...",
        "created": "2025-11-30"
      }
    ]
  }
}
```

#### 3. Версионирование онтологии

Журнал разработки должен фиксировать изменения онтологии:

```json
{
  "log": {
    "log-attributes": {
      "ontology_changes": [
        {
          "action": "added_class",
          "class": "feont:SignalSystem",
          "timestamp": "2025-11-30T10:30:00Z"
        }
      ]
    }
  }
}
```
